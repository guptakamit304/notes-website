Core Principle

Model-View-Controller(MVC)

Model
  • Manages the application dataand business logic.
  • Uses Eloquent ORM for database interactions.
View
  • Represents the UI of the application.
  • Uses the Blade templating engine for dynamic content.
Controller
  • Handles user requests and coordinates between the model and view.
  • Routes user actions to the appropriate logic and responses.
DRY(Don't Repeate Yourself)
  • Encourages code reuse and modularity
  • Avoids duplication by leveraging reusable components and services

Conversition over Configuration

  • Reduces the need for manual setup by adoping sensible defaults.
  • Simplifies the development process by elimating boilerplate code.
    • Eg:
    1. Default directory structure for Models View and Controllers
    2. Predefined configuration files for routing database connection and caching
The Root Dirrectory
  • app/: Contains application logic, such as models, controllers, and middleware.
  • bootstrap/: Initializes the application with app.php and caches configuration.
  • config/: Stores application configuration files.
  • database/: Manages migrations, seeders, and factories for the database.
  • public/: Web server entry point, containing index.php and public assets.
  • resources/: Holds views, language files, and frontend resources.
  • routes/: Defines web, API, and console routes for the application.
  • storage/: Stores logs, caches, and uploaded files.
  • tests/: Contains unit and feature test files.
  • vendor/: Houses third-party packages installed via Composer.
  • .env: Stores environment-specific variables like database credentials.
  • artisan: Command-line tool for Laravel tasks and development.
  • composer.json: Manages PHP dependencies for the application.
  • webpack.mix.js: Configures asset bundling with Laravel Mix.
